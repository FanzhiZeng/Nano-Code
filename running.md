# RUNNING

## 当前目标

让当前项目从“能跑通工具调用闭环的原型”继续推进到“具备基础任务规划能力的 coding agent”。

按照最新的 [README.md](/home/luo/project/nano_code/README.md) 和 [todo.md](/home/luo/project/nano_code/todo.md)，当前版本已经具备对话、工具调用、读写文件和基础网页搜索能力。下一轮工作的重点不再是补最小工具集，而是把 agent 的任务层、主控层和工具层边界继续做清楚。

## 当前代码状态

已经具备：

- `Anthropic` SDK 多轮对话
- `tool_use` / `tool_result` 循环
- 统一工具抽象、注册和分发
- `bash`、`read_file`、`write_file`、`web_search`
- 启动配置校验、日志落盘、异常兜底
- 前台按“模型中间输出 -> 工具状态 -> 最终回答”的顺序展示

当前主入口的大致职责边界是：

- [main.py](/home/luo/project/nano_code/main.py) 中的 `agent_loop()` 负责一次请求内的模型调用和工具回填循环
- [main.py](/home/luo/project/nano_code/main.py) 中的 `run()` 负责命令行输入和会话历史维护

目前主要短板：

- 还没有 `TodoManager`，agent 缺少显式任务规划能力
- `main.py` 仍然偏重，函数边界虽然比以前更清楚，但还没有真正拆出模块
- 工具层日志和错误表达还不够统一
- `bash`、`write_file`、`web_search` 的安全和错误处理仍然偏粗
- 没有测试，也没有上下文选择能力

## 本轮范围

本轮建议聚焦三件事：

1. 完成 `TodoManager`，为 agent 增加任务规划能力
2. 继续收口 [main.py](/home/luo/project/nano_code/main.py) 中的主循环和编排逻辑
3. 对齐工具层日志、错误表达和基础安全边界

本轮不做：

- 大规模模块重构
- 自动扫描整个项目后注入上下文
- 更复杂的文件编辑协议

## 具体事项

### 1. 完成 `TodoManager`

目标：

- 为 agent 提供显式任务列表和执行顺序
- 支持新增、更新、完成任务状态
- 让多步骤请求不再完全依赖模型隐式记忆

验收点：

- 用户提出多步骤任务时，程序内部能维护任务状态
- agent 可以根据任务进度决定下一步行动
- 任务规划信息不会破坏现有对话和工具调用闭环

### 2. 继续收口主循环

当前问题：

- `agent_loop()` 仍然同时处理模型请求、文本打印、工具状态输出、工具执行和 `tool_result` 回填
- `run()` 已经从命名上和 `agent_loop()` 分开，但主入口仍集中在一个文件里

要做的事：

- 继续提取“收集工具调用”的函数
- 继续提取“执行单个工具并生成 `tool_result`”的函数
- 给后续接入 `TodoManager` 留出更清晰的编排边界

验收点：

- `agent_loop()` 更短，职责更单一
- 接入任务规划逻辑时不需要继续堆在主循环里
- 工具失败不会导致进程退出

### 3. 补齐日志、错误表达和安全边界

目标：

- 工具调用开始时记录工具名和入参摘要
- 工具完成时记录结果长度、命中数或条目数
- 工具失败时记录明确错误
- 统一工具层返回格式，区分成功、失败和摘要信息
- 继续收紧 `bash`、`write_file`、`web_search` 的边界

验收点：

- 日志里能快速看出一次请求做了什么
- 工具失败时前台和日志都能给出可读信息
- 工具层行为更稳定，不依赖主循环做过多兜底

## 建议实施顺序

1. 先定义 `TodoManager` 的最小数据结构和状态流转
2. 再整理 `agent_loop()` 的内部边界
3. 然后统一工具层日志和返回格式
4. 最后补基础安全限制和错误分类

## 手动验证

建议至少验证下面这些场景：

1. 给一个多步骤请求，确认任务能够被拆解和跟踪
2. 问“读取 README.md”，确认现有 `read_file` 仍正常工作
3. 让模型执行需要写文件的任务，确认 `write_file` 仍可用
4. 给一个错误路径或异常输入，确认工具失败会回传模型而不是中断程序
5. 问一个需要 shell 的问题，确认 `bash` 仍然可用且日志完整
